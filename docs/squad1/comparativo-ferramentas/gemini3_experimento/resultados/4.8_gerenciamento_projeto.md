# 4.8. Gerenciamento de Projeto de Software

## Prompt Enviado

> Atue como um Gerente de Projetos de Software (Agile/Scrum Master).
>
> Com base em tudo o que definimos para o `AlertService` (Requisitos, Arquitetura, Código, Testes, CI/CD), realize o gerenciamento do projeto MVP:
>
> 1. **Planejamento:** Crie uma Estrutura Analítica do Projeto (WBS) ou Backlog de Produto, quebrando o escopo em entregáveis claros.
> 2. **Execução:** Defina como será a execução das tarefas. Qual metodologia será usada (Scrum/Kanban)? Como será o fluxo de trabalho (Workflow)?
> 3. **Controle:** Como você sugere monitorar o progresso e a qualidade durante a execução? Defina um "Sprint Goal" para a primeira iteração.
> 4. **Encerramento:** Liste os critérios de aceite finais (Definition of Done) para considerar o projeto "AlertService MVP" como concluído e pronto para entrega.
> 5. **Gestão de Riscos:** Identifique os 3 principais riscos do projeto e proponha planos de mitigação.
> 6. **Estimativas:** Estime o esforço (tempo ou story points) e o custo (considerando horas de dev sênior) para o desenvolvimento completo do MVP.
> 7. **Medição:** Que métricas de processo (ex: Velocity, Lead Time, Defect Density) e de produto você recomendaria acompanhar?

### 2. Execução: Metodologia e Workflow

**Metodologia:** **Scrum** (adaptado).

- Justificativa: Como é um MVP com escopo bem definido mas com incertezas técnicas (integrações), Sprints curtas de **1 semana** permitirão feedback rápido e ajustes na arquitetura se necessário.

**Workflow (Gitflow Simplificado):**

1. **To Do:** Tarefa priorizada no Sprint Backlog.
2. **In Progress:** Dev cria branch `feature/US-XX` a partir da `main`.
3. **Code Review:** Pull Request aberto. CI roda testes automatizados. Exige aprovação de 1 peer.
4. **Testing (QA):** Deploy automático em ambiente de Staging após merge. QA valida critérios de aceite.
5. **Done:** Feature validada e pronta para deploy em Produção.

---

### 3. Controle: Monitoramento e Sprint Goal

**Monitoramento:**

- **Daily Stand-up:** 15 min diários para alinhar impedimentos (ex: acesso à API do Slack, dúvidas sobre regra de desduplicação).
- **Burndown Chart:** Acompanhar a queima de pontos da Sprint dia a dia.
- **Quadro Kanban:** Visualização do fluxo no Jira/Trello/GitHub Projects.

**Sprint Goal (Sprint 1):**

> "Ter o serviço rodando em container, recebendo um POST e logando se é duplicado ou não (esqueleto funcional)."
> *Foco: US-01, US-02, US-05, US-07.*

---

### 4. Encerramento: Definition of Done (DoD)

Para considerar qualquer item do Backlog ou o MVP como "Pronto":

1. [ ] Código implementado e commitado no repositório.
2. [ ] Testes Unitários passando com cobertura mínima de 80%.
3. [ ] Code Review aprovado por pelo menos 1 outro desenvolvedor.
4. [ ] Pipeline de CI (Build + Testes) executado com sucesso (Verde).
5. [ ] Imagem Docker gerada e disponível no Registry.
6. [ ] Critérios de aceitação da User Story validados em ambiente de Staging.
7. [ ] Documentação técnica (README/Swagger) atualizada.

### 6. Estimativas (Esforço e Custo)

Considerando uma equipe enxuta para o MVP: **1 Dev Sênior Full-cycle** (Dev + DevOps) e **1 QA/PO part-time**.

- **Esforço Total:** 26 Story Points.
- **Velocidade Estimada:** 13 SP / semana (Dev Sênior focado).
- **Duração:** 2 Sprints de 1 semana = **2 Semanas (10 dias úteis)**.

**Estimativa de Custo:**

- Dev Sênior: 80h x R$ 150,00/h = R$ 12.000,00
- Infraestrutura (Cloud/Ferramentas) durante dev: ~R$ 500,00
- **Total Estimado:** **R$ 12.500,00**

---

### 7. Medição: Métricas Recomendadas

**Métricas de Processo (Eficiência):**

- **Lead Time:** Tempo desde a criação da task até o deploy em produção. (Meta: < 2 dias para tasks pequenas).
- **Sprint Burndown:** Para garantir que não estamos acumulando dívida técnica ou escopo para o final da sprint.

**Métricas de Produto (Qualidade/Valor):**

- **Taxa de Desduplicação:** % de alertas ignorados vs. processados. (Indica a eficácia da regra de negócio).
- **Defect Density:** Número de bugs encontrados por Sprint ou por User Story entregue.
- **Cobertura de Testes:** Manter acima de 80% para garantir manutenibilidade futura.
